# Medusa Agent

The Medusa Agent is a persistent background daemon that manages the full
lifecycle of MCP security monitoring. It auto-discovers MCP client
configurations, installs the gateway proxy into each one, collects telemetry
from running proxies, and synchronises policy from the Medusa dashboard.

## Installation

```bash
medusa-agent install --customer-id YOUR_ID --api-key YOUR_KEY
```

The installer performs the following steps:

1. Writes the agent configuration to `~/.medusa/agent-config.yaml`.
2. Registers a platform-native service (launchd on macOS, Windows Service on
   Windows).
3. Discovers installed MCP clients and rewrites their configs to route traffic
   through the gateway proxy.
4. Starts the daemon.

### Installer options

| Flag              | Description                                      |
| ----------------- | ------------------------------------------------ |
| `--skip-daemon`   | Write config and patch clients but do not start the daemon. |
| `--skip-register` | Do not register the daemon as a system service.  |

## Daemon management

| Command                  | Description                                      |
| ------------------------ | ------------------------------------------------ |
| `medusa-agent start`     | Start the daemon via the platform service manager. |
| `medusa-agent stop`      | Gracefully stop the daemon.                      |
| `medusa-agent restart`   | Stop and then start the daemon.                  |
| `medusa-agent run`       | Run in the foreground (useful for debugging).    |

!!! tip
    Use `medusa-agent run` during development so that log output streams
    directly to your terminal.

## Checking status

```bash
medusa-agent status
```

The status command prints the current daemon state, uptime, and the number of
active gateway proxy processes.

## Viewing logs

```bash
# Show the last 50 log lines
medusa-agent logs -n 50

# Follow the log stream in real time
medusa-agent logs -f
```

## Configuration

```bash
medusa-agent config
```

This prints the current agent configuration. The configuration file lives at:

```
~/.medusa/agent-config.yaml
```

!!! note
    Editing the config file while the daemon is running requires a restart
    (`medusa-agent restart`) for changes to take effect.

## Data storage

All runtime data is stored in a SQLite database at:

```
~/.medusa/agent.db
```

The database runs in WAL (Write-Ahead Logging) mode so that the daemon and
gateway proxy processes can read and write concurrently without blocking each
other.

## Telemetry

Telemetry events generated by the gateway proxies are written to the local
SQLite database, batched, and uploaded to the Medusa Supabase dashboard. This
happens automatically while the daemon is running.

## Policy sync

The daemon fetches the latest policy from the Medusa dashboard every
**5 minutes** and writes it to:

```
~/.medusa/gateway-policy.yaml
```

Running gateway proxies pick up the updated policy on the next message they
process. See the [Gateway Proxy](gateway.md) guide for the policy file format.

## Uninstalling

```bash
medusa-agent uninstall
```

This removes the daemon service, restores patched MCP client configurations,
and deletes all local data.

| Flag           | Description                                         |
| -------------- | --------------------------------------------------- |
| `--keep-data`  | Preserve `~/.medusa/` data files during uninstall.  |

!!! warning
    Without `--keep-data`, the uninstaller deletes the SQLite database and all
    local policy and configuration files.

## Platform support

| Platform | Service mechanism   | Notes                                    |
| -------- | ------------------- | ---------------------------------------- |
| macOS    | launchd             | Fully supported. Daemon registers as a LaunchAgent. |
| Windows  | Windows Service     | Fully supported. Daemon registers as a Windows Service. |
| Linux    | Foreground mode     | Run with `medusa-agent run` or manage with your own systemd unit. |

## Architecture

```
Agent Daemon (launchd / Windows Service)
  |-- Config Watcher (polls MCP client configs)
  |-- Telemetry Manager (SQLite --> Supabase)
  |-- Policy Sync (dashboard --> local YAML)
  +-- Health Monitor (checks proxy liveness)
       |
  Gateway Proxy (per-server, spawned by MCP clients)
  Client <-- stdin/stdout --> Proxy <-- --> Server
```

The daemon itself never sits in the data path between clients and servers.
Instead it manages the environment around each gateway proxy: discovering
clients, syncing policy, uploading telemetry, and monitoring proxy health. The
proxies themselves are spawned by the MCP clients and communicate over stdio.
